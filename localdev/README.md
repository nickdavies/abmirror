# Local Development Environment

Run Actual Budget in Docker and bootstrap from your exported budget zips for testing ab-mirror pipelines.

## Prerequisites

- Docker (or docker compose)
- Node.js 22+
- 3 Actual Budget export files (.zip from the UI)

## Server

- **URL:** `http://localhost:5007`
- **Password:** `test`

## Quick Start

```bash
# 1. Bootstrap from 3 zips (starts Docker, imports, writes config)
# Zip paths are relative to your current directory, or use absolute paths
cd localdev && ./bootstrap.sh nick.zip britta.zip joint.zip

# Encrypted exports are uploaded unencrypted (export contains decrypted data).
# No keys needed for bootstrap.

# 2. Run a pipeline
./run-pipeline.sh --pipeline pipelines/split-only.yaml
./run-pipeline.sh --pipeline my-experiment.yaml --dry-run
```

## Commands

| Command | Description |
|---------|-------------|
| `./bootstrap.sh [--port 5007] zip1 zip2 zip3` | Start Docker, import zips, write `config.yaml` |
| `./teardown.sh [--clean]` | Stop Docker containers; use `--clean` to also remove `./data` and `config.yaml` |
| `./run-pipeline.sh --pipeline FILE \| -` | Merge pipeline with config, run ab-mirror |
| `./run-pipeline.sh --config PATH` | Use a different base config (default: `config.yaml`) |
| `./run-pipeline.sh --dry-run` | Simulate without writing |
| `./run-pipeline.sh --step N` | Run only pipeline step N (1-based) |

## File Layout

- `docker-compose.yml` - Actual server on port 5007, persistent `./data`
- `bootstrap.sh` - Entry script
- `teardown.sh` - Stop containers (optionally `--clean` to remove data and config)
- `bootstrap.ts` - Imports zips via Actual API
- `run-pipeline.sh` - Merges pipeline YAML, runs ab-mirror
- `snapshot.ts` - Captures budget snapshots as normalized YAML fixtures (see below)
- `config.yaml` - Generated by bootstrap (gitignored)
- `pipelines/` - Example pipeline snippets

## Generating integration test fixtures

The `snapshot.ts` script connects to the running Actual server and exports a
normalized fixture file suitable for use as `before.yaml` / `after.yaml` in the
automated integration tests (`test/integration/cases/`).

### Workflow

```bash
cd localdev

# 1. Bootstrap and shape your budgets
./bootstrap.sh nick.zip britta.zip joint.zip
# (optionally make manual edits in the Actual UI at http://localhost:5007)

# 2. Capture the before state
npx tsx snapshot.ts --out before.yaml

# 3. Run the pipeline you want to test
./run-pipeline.sh --pipeline pipelines/my-pipeline.yaml

# 4. Capture the after state
npx tsx snapshot.ts --out after.yaml

# 5. Add as an automated test case
CASE=my-case
mkdir -p ../test/integration/cases/$CASE
cp before.yaml after.yaml ../test/integration/cases/$CASE/
cp pipelines/my-pipeline.yaml ../test/integration/cases/$CASE/pipeline.yaml

# 6. Confirm it passes
cd .. && npm test
```

### Snapshot CLI options

```
npx tsx snapshot.ts [--config PATH] [--budgets alias1,alias2] --out PATH
```

| Option | Default | Description |
|--------|---------|-------------|
| `--config PATH` | `./config.yaml` | Base config to read server URL and budget sync IDs from |
| `--budgets a,b` | all in config | Comma-separated list of budget aliases to include |
| `--out PATH` | (required) | Output YAML file |

### How normalization works

Transaction UUIDs and internal budget IDs are not stable across server restarts
or re-imports. The snapshot CLI normalizes them to stable placeholders:

- **Transaction IDs** — all visible transactions are sorted globally by
  `(budgetAlias, accountName, date, notes, payee_name)` then by amount, and
  assigned sequential IDs: `TX-1`, `TX-2`, …  Subtransactions within a parent
  `TX-N` are numbered `TX-N-SUB-1`, `TX-N-SUB-2`, …
- **`imported_id` values** — `ABMirror:<localBudgetId>:<txUuid>` is rewritten to
  `ABMirror:<alias>:<TX-N>` using the maps built above.  References to unknown
  budget IDs are left as-is.

The same algorithm is used by `exportRuntimeToFixture` in the test harness, so
the two outputs are directly comparable.

See [`test/integration/cases/README.md`](../test/integration/cases/README.md) for full fixture format documentation.
